# Data Structures

## Inputs

### `Anchor`
Where a cable connects. Always has a position; optionally has a direction hint for exit angle.

```rust
pub struct Anchor {
    pub position: Vec3,
    pub direction: Option<Vec3>,  // Preferred exit direction (for smooth tangents at connections)
}
```

### `Obstacle`
An AABB with a transform. The planner routes around these.

```rust
pub struct Obstacle {
    pub half_extents: Vec3,  // Local-space AABB half-extents
    pub position: Vec3,
    pub rotation: Quat,
}
```

Why `half_extents` + `position` + `rotation` instead of `min`/`max`? Because objects in the 3D world have transforms. The planner must transform the AABB into world space for intersection tests. This matches how Bevy represents colliders.

### `RouteRequest`
Everything a solver needs to compute a route.

```rust
pub struct RouteRequest<'a> {
    pub start: Vec3,
    pub end: Vec3,
    pub obstacles: &'a [Obstacle],
    pub resolution: u32,  // Sample points per segment
}
```

Deliberately minimal. Algorithm-specific parameters live on the solver structs themselves (e.g., `CatenarySolver.slack`), not on the request. The request is what varies per-cable; the solver config is what varies per-algorithm.

## Outputs

### `CableSegment`
The geometry of one continuous curve between two waypoints.

```rust
pub struct CableSegment {
    pub points: Vec<Vec3>,       // Sampled positions along the curve
    pub tangents: Vec<Vec3>,     // Unit tangent at each point
    pub arc_lengths: Vec<f32>,   // Cumulative arc length at each point
    pub length: f32,             // Total arc length of this segment
}
```

Why tangents? Mesh generation needs them for orienting tube cross-sections. Running light effects need them for flow direction. They're cheap to compute during sampling and expensive to reconstruct later.

Why arc lengths? UV mapping, LOD decisions, running light animation, and oscilloscope-style visualization all need to know "how far along the cable am I?" Cumulative arc length answers that in O(1) per point.

### `CableGeometry`
The full route — all segments concatenated. This is the render-agnostic output, analogous to Clay's `RenderCommand` array.

```rust
pub struct CableGeometry {
    pub segments: Vec<CableSegment>,
    pub total_length: f32,
    pub waypoints: Vec<Vec3>,     // Structural points (start, bends, end)
}
```

Segments are kept separate (not flattened into one big `Vec<Vec3>`) because:
- Each segment may have been generated by a different algorithm
- Waypoint boundaries are useful for rendering (e.g., visible joint at a bend)
- Flat iteration is trivial: `geometry.segments.iter().flat_map(|s| &s.points)`

`waypoints` stores the structural points for debug visualization and hit testing. These are the planner's output before curve filling.

## Solver Configurations

### `CatenarySolver`
```rust
pub struct CatenarySolver {
    pub slack: f32,        // Cable length / straight-line distance. 1.0 = taut, 1.5 = very loose
    pub gravity: Vec3,     // Gravity direction and magnitude (typically Vec3::NEG_Y * 9.81)
    pub resolution: u32,   // Default sample count per segment
}
```

### `AStarPlanner`
```rust
pub struct AStarPlanner {
    pub grid_size: f32,    // Voxel size for the 3D search grid
    pub margin: f32,       // Clearance around obstacles
}
```

### `OrthogonalPlanner`
```rust
pub struct OrthogonalPlanner {
    pub snap_angles: Vec<f32>,    // Allowed angles in radians (e.g., [0, PI/2, PI, 3*PI/2])
    pub bend_radius: f32,         // Minimum distance from obstacle before bending
}
```

## Bevy Components (Plugin Layer)

### `Cable`
The user-facing component. Declares intent.

```rust
#[derive(Component)]
#[require(ComputedCableGeometry, Transform)]
pub struct Cable {
    pub start: Anchor,
    pub end: Anchor,
    pub solver: Box<dyn RouteSolver>,
    pub obstacles: Vec<Obstacle>,   // Could also be sourced from spatial queries
}
```

### `ComputedCableGeometry`
Auto-populated by the `compute_cable_routes` system. Rendering reads from this.

```rust
#[derive(Component, Default)]
pub struct ComputedCableGeometry {
    pub geometry: Option<CableGeometry>,
}
```

The system uses `Changed<Cable>` to avoid recomputing unchanged routes.
